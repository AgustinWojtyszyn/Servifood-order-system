P0 Critico
----------

1. Archivado manual masivo archiva pedidos pendientes de cualquier fecha

Archivos:
- src/hooks/useDailyOrdersData.js, linea aproximada 160
- src/services/orders/ordersService.js, linea aproximada 6
- supabase/migrations/20260226_archive_orders_bulk.sql, linea aproximada 25

Riesgo real:
Desde /daily-orders, el boton de "archivar todos" muestra pedidos de la fecha operativa,
pero el RPC actual hace:

UPDATE public.orders
SET status = 'archived'
WHERE status = ANY(statuses)

Eso archiva todos los pending de la base, incluyendo pedidos futuros o de otras fechas.

Como reproducir:
1. Crear un pedido pending para delivery_date = 2026-06-25.
2. Crear otro pending para delivery_date = 2026-06-26.
3. Entrar a /daily-orders el 24/06/2026.
4. Ejecutar "Archivar todos".
5. Ambos pedidos quedan archived.

Fix recomendado:
Hacer el archivado masivo acotado por delivery_date, no global.
Idealmente agregar o cambiar el RPC admin a algo como:

archive_orders_bulk_by_delivery_date(p_delivery_date date, statuses text[])

Luego, desde /daily-orders, pasar la fecha operativa.

Requiere SQL:
Si.

P1 Alto
-------

1. mode: send registra el reporte pero no archiva pedidos

Archivos:
- supabase/functions/daily-orders-report/index.ts, linea aproximada 480
- supabase/functions/_shared/daily_report.ts, linea aproximada 467

Riesgo real:
El helper indica que send debe archivar, y el test lo espera conceptualmente, pero index.ts
no ejecuta ningun update ni RPC de archivado despues de enviar el email.

El cron puede mandar el reporte, registrar daily_report_runs como sent y dejar todos los
pedidos como pending.

Como reproducir:
1. Ejecutar la Edge Function con mode: send para una fecha con pedidos pending.
2. Verificar daily_report_runs: queda sent.
3. Verificar orders: los pedidos siguen pending.

Fix recomendado:
Despues de enviar el email correctamente, archivar solo:

status = 'pending'
delivery_date = reportDate

Conviene reutilizar el mismo RPC acotado por fecha del P0.

Requiere SQL:
Si se implementa por RPC. No estrictamente si la Edge Function usa service role para update directo.

2. Un run en estado running puede bloquear reportes futuros de esa fecha

Archivos:
- supabase/functions/daily-orders-report/index.ts, linea aproximada 318
- supabase/functions/_shared/daily_report.ts, linea aproximada 475
- supabase/migrations/20260623090500_daily_report_runs.sql, linea aproximada 1

Riesgo real:
Si la funcion muere despues de insertar daily_report_runs.status = 'running' y antes del catch
o del upsert a failed, el proximo cron salta el envio porque running se considera procesado.

Eso puede dejar un dia sin reporte automatico.

Como reproducir:
1. Forzar interrupcion despues de insertar el lock running.
2. Volver a ejecutar mode: send.
3. La funcion responde skipped con status running.

Fix recomendado:
Tratar running viejo como stale lock. Por ejemplo, si created_at es anterior a 30 minutos,
permitir reintento o marcar failed.

Requiere SQL:
Probablemente no, porque daily_report_runs ya tiene created_at.

3. /daily-orders considera pendiente cualquier estado distinto de archived

Archivos:
- src/hooks/useDailyOrdersFilters.js, linea aproximada 39
- src/components/DailyOrders.jsx, linea aproximada 85

Riesgo real:
Si existe un pedido cancelled para la fecha operativa, aparece y se exporta como pendiente
en /daily-orders, WhatsApp y Excel, porque el filtro usa:

order.status !== 'archived'

Como reproducir:
1. Crear un pedido cancelled con delivery_date operativo.
2. Abrir /daily-orders con filtro Pendientes.
3. El pedido aparece en el resumen/exportacion.

Fix recomendado:
Para el filtro pending, usar estrictamente:

order.status === 'pending'

Requiere SQL:
No.
