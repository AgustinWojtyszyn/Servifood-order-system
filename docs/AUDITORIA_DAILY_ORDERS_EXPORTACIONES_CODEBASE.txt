Mapa de funciones/componentes involucrados
-----------------------------------------
DailyOrders:
Compone el bloque operativo. Usa useDailyOrdersData como fuente base, useDailyOrdersFilters para sortedOrders y dispara exportaciones usando sortedOrders.

useDailyOrdersData:
Lee todos los pedidos de orders_with_person_key disponibles para el admin, filtra por delivery_date == getTomorrowISOInTimeZone(), enriquece user_name/user_email y calcula stats.

useDailyOrdersFilters:
Filtra en memoria por location, status, dish, side y orden. availableSides sale de getCustomSideFromResponses.

calculateStats:
Calcula total, byLocation, byDish, totalItems, archived y pending sobre la lista ya filtrada por delivery_date.

buildOperationalSummary:
Calcula resumen visible de platillos, guarniciones y bebidas sobre sortedOrders.

buildLocationCards:
Construye resumen por ubicacion sobre allOrders.

buildPrintStats:
Calcula turnos, guarniciones y opciones para print summary.

filterOrdersByCompany:
Filtra por location o company/company_slug/target_company. En la practica, el flujo revisado suele depender de location.

buildDailyOrdersSummary:
Construye resumen central para Excel y texto operativo con totales, byLocation, byMenu, byLocationMenu, adicionales, comentarios, servicios e inconsistencias.

formatDailyOrdersForWhatsApp:
Construye texto compacto agrupado por ubicacion/menu y asocia guarniciones a menu cuando puede.

exportDailyOrdersExcel:
Filtra por exportCompany, deduplica por id, genera workbook y descarga.

exportDailyOrdersPdf:
Genera HTML imprimible desde sortedOrders; usa buildOrderPreview y buildTurnSummary.

shareDailyOrdersWhatsApp:
Usa formatDailyOrdersForWhatsApp(sortedOrders) y abre WhatsApp.

getCustomSideFromResponses:
Extrae la primera respuesta cuyo title contiene guarn.

extractCustomResponses:
Separa custom_responses en side, beverage y additional.

getSideValuesFromOrder / findItemForSide:
Intentan asociar guarniciones a menu usando metadata de item_id/menu_id/slotIndex; si no hay identidad, se asocia al primer item.

archivePendingOrdersByDeliveryDate:

archive_orders_bulk_by_delivery_date:
Archiva solo orders con delivery_date indicado y status dentro de p_statuses.

Flujo paso a paso de /daily-orders
----------------------------------
1. Usuario abre /daily-orders.
2. DailyOrders monta useDailyOrdersData(user).
3. useDailyOrdersData valida si user es admin con usersService.getUserById.
4. Si es admin, fetchDailyOrders lee db.getOrdersWithPersonKey().
5. getOrdersWithPersonKey consulta orders_with_person_key ordenado por created_at desc.
6. useDailyOrdersData lee db.getAdminPeopleUnified() para enriquecer nombres/emails por person_key.
7. Calcula operationalDate = getTomorrowISOInTimeZone().
8. Filtra pedidos por String(order.delivery_date) === operationalDate. No usa created_at para decidir inclusion operativa.
9. Mapea user_name/user_email desde customer fields o admin_people_unified.
10. Construye availableDishes desde order.items.
11. Calcula stats sobre todayOrders con calculateStats.
12. useDailyOrdersFilters recibe orders y aplica filtros de UI: ubicacion, status, platillo, guarnicion y orden.
13. DailyHeader muestra stats filtradas por status para metricas de cabecera.
14. DailySummary muestra resumen operativo sobre sortedOrders y tarjetas por ubicacion sobre allOrders.
15. DailyOrdersTable renderiza sortedOrders.
16. Acciones de exportacion usan sortedOrders.
17. Archivado individual usa updateOrderStatus(order.id, 'archived').
18. Archivado masivo usa archivePendingOrdersByDeliveryDate({ deliveryDate: operationalDate, statuses: ['pending'] }).

Flujo de exportacion manual
---------------------------
WhatsApp:
- DailyOrders llama shareDailyOrdersWhatsApp(sortedOrders, selectedStatus).
- sortedOrders ya trae filtros de ubicacion/status/platillo/guarnicion/orden.
- shareDailyOrdersWhatsApp llama formatDailyOrdersForWhatsApp(sortedOrders).
- formatDailyOrdersForWhatsApp agrupa por ubicacion, luego por menu, y muestra guarniciones bajo menu.
- No aplica exportCompany. Usa exactamente sortedOrders visibles por filtros principales.

Excel:
- DailyOrders llama exportDailyOrdersExcel({ sortedOrders, exportCompany, selectedLocation, selectedStatus, stats }).
- exportDailyOrdersExcel aplica filterOrdersByCompany(sortedOrders, exportCompany).
- Deduplica por order.id.
- buildDailyOrdersSummary arma Resumen, Detalle, Comentarios e Inconsistencias.
- Usa fuente filteredOrders derivada de sortedOrders, por lo que respeta filtros principales y ademas el selector Empresa para exportar.

PDF:
- DailyOrders llama exportDailyOrdersPdf(sortedOrders).
- Usa sortedOrders directamente.
- Genera HTML con resumen por turno, empresas por turno y detalle.
- No aplica exportCompany.
- Usa getTomorrowDate() para label de entrega, no el delivery_date del primer pedido.

Fuente de verdad:
- Pantalla, WhatsApp, Excel y PDF parten de sortedOrders.
- Excel agrega filtro extra exportCompany.
- WhatsApp y PDF no usan exportCompany.
- La logica de formateo no es unica: pantalla usa dailyOrderCalculations/DailyOrderRow, Excel usa dailyOrdersExportModel, WhatsApp usa buildWhatsAppLocationMenuSummary, PDF usa buildOrderPreview/buildTurnSummary. Hay normalizacion compartida parcial, pero hay logica duplicada.

Flujo de guarniciones/custom_responses
--------------------------------------
Lectura:
- Los pedidos traen custom_responses desde orders_with_person_key.
- normalizeOrderForReadOnly parsea custom_responses si viene string JSON o array.

Interpretacion en pantalla:
- getCustomSideFromResponses busca responses cuyo title contiene guarn y devuelve answer/response.
- DailyOrderRow muestra "Guarnicion: ..." en tabla/card.
- availableSides usa getCustomSideFromResponses para poblar el filtro de guarnicion.
- buildOperationalSummary y buildLocationCards cuentan guarniciones por respuesta.

Interpretacion en Excel:
- extractCustomResponses separa side, beverage y additional.
- buildDailyOrdersExcelDetailRow pone Guarniciones, Respuestas personalizadas y otros campos.
- buildDailyOrdersSummary incluye guarniciones/adicionales por ubicacion.

Interpretacion en WhatsApp:
- getSideValuesFromOrder extrae guarniciones.
- findItemForSide intenta asociarlas al item correcto usando metadata si existe.
- Si no hay identidad suficiente, asocia la guarnicion al primer item del pedido.

Interpretacion en PDF:
- buildOrderPreview muestra itemsText y optionsText.
- La guarnicion se agrega a itemsText; otras opciones van a optionsText.

Asociacion menu-guarnicion:
- El modelo soporta asociacion por identidad si custom_responses trae item_id/menu_id/slotIndex o nombres.
- Si esa metadata no viene en el pedido, WhatsApp cae al primer item. En pedidos con un solo menu no es problematico; en pedidos con multiples items puede asociar guarnicion al menu equivocado.

Validaciones existentes
-----------------------
- /daily-orders filtra por delivery_date operativo, no por created_at.
- El default de selectedStatus es pending.
- Se puede ver pending, archived o all.
- Archivado masivo usa delivery_date y statuses ['pending'].
- RPC legacy archive_orders_bulk queda deshabilitado y fuerza usar delivery_date.
- Archivado individual solo se ofrece si order.status === 'pending'.
- Excel deduplica por id antes de exportar.
- Excel agrega hoja de inconsistencias.
- normalizeOrderForReadOnly evita romper si items/custom_responses llegan como string JSON.
- availableSides sale de pedidos reales cargados.
- Filtros de pantalla y exportaciones parten de sortedOrders.
- COMPANY_LOCATIONS centraliza Ccp, La Laja, Padre Bueno, Los Berros, Genneia y Genneia O&M.

Validaciones faltantes o dudosas
--------------------------------
1. useDailyOrdersData trae todos los pedidos de orders_with_person_key y filtra delivery_date en cliente.
   Con 9770 orders actuales funciona, pero no escala idealmente y depende de permisos/limites de PostgREST.

2. PDF usa getTomorrowDate() para el label de entrega.
   Como sortedOrders ya fue filtrado por delivery_date operativo, normalmente coincide. Pero el PDF no toma el delivery_date real de los pedidos; si se exportara una lista distinta en el futuro, podria rotular mal.

3. WhatsApp y PDF no usan exportCompany.
   Excel puede exportar una empresa distinta usando selector Empresa (para exportar). WhatsApp/PDF exportan sortedOrders, no exportCompany. Esto puede generar diferencias si el operador espera que ese selector aplique a todos los botones.

4. Asociacion guarnicion-menu depende de metadata opcional.
   Si custom_responses no trae item_id/menu_id/slotIndex, WhatsApp asigna al primer item.

5. Agrupacion por empresa depende principalmente de location.
   filterOrdersByCompany tambien revisa company/company_slug/target_company, pero el flujo principal de creacion guardaba location. Si location no identifica univocamente la empresa, puede haber ambiguedad.

6. Hay varias logicas de resumen/formato.
   Pantalla, Excel, WhatsApp y PDF comparten algunos helpers, pero no una unica fuente completa. Esto aumenta riesgo de diferencias.

7. getTomorrowDate usa new Date local y toLocaleDateString es-ES.
   El filtro critico usa getTomorrowISOInTimeZone, pero labels visuales/PDF usan helper distinto. Puede haber diferencias de etiqueta cerca de medianoche o por zona horaria del navegador.

8. Filtro de platillo compara item.name exacto.
   availableDishes se arma desde item.name sin normalizar como normalizeDishName. Puede haber diferencias si pantalla muestra "Plato Principal" pero filtro usa nombre raw.

Riesgos P0
----------
No se detectaron P0 evidentes en el alcance revisado.

No se observo un camino claro para mezclar pedidos de otra fecha en /daily-orders: la carga filtra por delivery_date operativo. El archivado masivo tambien se limita por delivery_date y pending. Las exportaciones parten de sortedOrders, que ya viene de la lista filtrada por fecha operativa.

Riesgos P1
----------
1. Diferencia entre Excel vs WhatsApp/PDF por exportCompany.
   Excel respeta el selector Empresa (para exportar), pero WhatsApp y PDF no. Riesgo operativo: enviar por WhatsApp/PDF mas pedidos que los incluidos en Excel si el operador cree que exportCompany aplica a todo.

2. PDF etiqueta entrega con getTomorrowDate, no con delivery_date del dataset.
   Riesgo de etiqueta incorrecta en edge cases o si se reutiliza exportDailyOrdersPdf con datos no operativos.

3. Guarnicion puede asociarse al menu equivocado en WhatsApp.
   Si no hay metadata de identidad en custom_responses y hay multiples items, findItemForSide usa el primer item.

4. Ambiguedad empresa/ubicacion.
   Los reportes agrupan mayormente por location. Si se necesitan empresas separadas con misma location o nombres historicos, puede haber mezcla conceptual.

5. Diferencias de logica entre pantalla, Excel, WhatsApp y PDF.
   Pantalla usa DailyOrderRow/buildOrderPreview, Excel usa dailyOrdersExportModel, WhatsApp usa buildWhatsAppLocationMenuSummary y PDF usa otro HTML. Pueden divergir en casos de cena, bebidas, guarniciones y nombres normalizados.

6. Filtro client-side sobre todos los pedidos.
   No es un error de consistencia hoy, pero con mas datos podria degradar o depender de limites de API. Si PostgREST limita filas, podria ocultar pedidos del delivery_date esperado.

Riesgos P2
----------
1. Nombres confusos.
   Variables como todayOrders representan la fecha operativa de entrega, no "hoy" calendario.

2. Helpers de fecha duplicados.
   getTomorrowISOInTimeZone se usa para filtro real; getTomorrowDate se usa para labels. Conviene alinear semantica.

3. Logica duplicada de custom_responses.
   getCustomSideFromResponses, extractCustomResponses, getCustomResponsesTextForExcel y buildOrderPreview interpretan respuestas de formas parecidas pero no identicas.

4. WhatsApp tiene dos modelos de texto.
   Existe formatDailyOrdersOperationalText y formatDailyOrdersForWhatsApp; el flujo actual usa el segundo, mas compacto.

5. Status legacy limitado.
   Filtros solo contemplan pending/archived/all; getStatusText contempla cancelled/unknown. No se detectaron estados raros en DB, pero conviene mantener pruebas.

6. Detail UI muestra hora created_at.
   Correcto como hora de pedido, pero puede confundirse con criterio operativo si no se rotula claramente.

Conclusion
----------
El bloque parece parcialmente solido a solido.

La base critica esta bien: /daily-orders filtra por delivery_date operativo, los filtros de status se aplican sobre esa lista, y el archivado masivo usa delivery_date + pending en RPC. Las exportaciones parten de sortedOrders, por lo que no deberian mezclar fechas si la pantalla esta correcta.

No se detectaron P0 evidentes. Los riesgos principales son P1: diferencias entre Excel y WhatsApp/PDF por exportCompany, asociacion imperfecta de guarniciones a menu cuando falta metadata, labels de PDF basados en getTomorrowDate y dependencia fuerte de location como empresa/ubicacion.

Proximos pasos recomendados
---------------------------
1. Auditar guarniciones con datos reales de orders.custom_responses e items para confirmar metadata de item/menu/slotIndex.
2. Auditar empresas/ubicaciones y decidir si orders debe persistir company_slug/company_name ademas de location.
3. Auditar autoexportado daily-orders-report y compararlo contra exportaciones manuales.
4. Auditar permisos/RLS de orders_with_person_key y admin_people_unified.
5. Crear fixtures de pedidos representativos para comparar pantalla, Excel, WhatsApp y PDF.
